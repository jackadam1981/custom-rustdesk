# 安全措施说明

## 架构设计

本项目采用双工作流架构来保护敏感信息：

### 1. 工作流架构

#### **IssuesBuildRustdesk.yml** - Issues 处理工作流
- **触发**：Issue 创建/编辑/标签
- **功能**：
  - 立即检测和清理敏感信息
  - 解析 Issue 中的参数
  - 触发主构建工作流
  - 添加安全通知和标签

#### **CustomBuildRustdesk.yml** - 主构建工作流
- **触发**：仅手动触发
- **功能**：
  - 直接使用传入的参数
  - 并发控制（只允许一个实例）
  - 执行完整的构建流程

### 2. 敏感信息处理

#### **立即清理机制**
- **自动检测**：Issue 创建或编辑时立即检测敏感信息
- **即时清理**：检测到敏感信息后立即清理，无需等待构建队列
- **安全通知**：清理后自动添加安全警告和说明

#### **受保护的敏感参数**
以下参数会被自动检测和清理：

- `--super_password`：超级密码
- `--password` / `--passwd`：密码
- `--rs_pub_key`：公钥
- `--private_key` / `--secret_key`：私钥/密钥
- `--api_server`：API 服务器地址
- `--rendezvous_server`：服务器地址
- `--server`：服务器地址
- `--token`：令牌
- `--api_key` / `--access_key`：API 密钥

### 3. 工作流程

#### **Issue 触发流程：**
1. **Issue 创建** → 立即触发 IssuesBuildRustdesk.yml
2. **敏感信息检测** → 自动识别敏感参数
3. **即时清理** → 替换为 `[已清理]`
4. **参数解析** → 解析 Issue 中的参数
5. **触发构建** → 调用 CustomBuildRustdesk.yml
6. **构建执行** → 使用解析的参数进行构建

#### **手动触发流程：**
1. **手动触发** → 直接使用手动输入的参数
2. **构建执行** → 使用手动设置的参数进行构建
3. **并发控制** → 只允许一个实例运行

### 4. 安全建议

#### ✅ 推荐做法：
- 使用配置预设（如：CONFIG_A, CONFIG_B）
- 联系管理员设置 Secrets
- 使用私有仓库进行敏感操作
- 在文档中说明安全要求

#### ❌ 避免做法：
- 在公开 Issue 中直接输入敏感信息
- 使用明文密码或密钥
- 在代码中硬编码敏感信息

### 5. 自动化处理

- **无需人工干预**：Issues 创建后自动处理
- **无需安全标签**：系统自动清理敏感信息
- **无需管理员审核**：完全自动化流程

### 6. 并发控制

#### **IssuesBuildRustdesk.yml**
- 并发组：`issues-build-IssuesBuildRustdesk-main`
- 策略：不取消正在运行的工作流
- 功能：自动处理，无需人工干预

#### **CustomBuildRustdesk.yml**
- 并发组：`custom-build-main`
- 策略：不取消正在运行的工作流
- 效果：只允许一个构建实例运行

### 7. 错误处理

- **触发失败**：自动添加错误通知和解决方案
- **权限问题**：提供手动触发指导
- **网络问题**：重试机制和超时处理

### 8. 联系方式

如有安全相关问题，请联系项目维护者。
